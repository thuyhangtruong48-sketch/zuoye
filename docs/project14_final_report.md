# 项目14：避开危险路段的救援路径规划最终报告

## 一、项目背景

地震、暴雨、城市内涝等灾害发生后，道路通行条件会发生变化。部分道路可能处于灾害影响区，出现塌方、积水、拥堵或通行效率下降。传统导航通常以距离最短或时间最短为目标，但救援车辆更需要考虑道路安全性和通行可靠性。

本项目使用真实地图道路数据和公开历史灾害资料，构建道路网络模型，识别受灾影响路段，并使用 Dijkstra 算法分别计算普通最短路径和安全路径。

## 二、项目目标

1. 使用高德 Web 服务获取真实道路路线数据。
2. 使用公开历史灾害资料构建地震和洪水两个场景。
3. 通过空间叠加方法将灾害影响区映射到道路网络。
4. 将塌方、积水、拥堵等因素转化为道路权重。
5. 使用 Dijkstra 算法计算普通最短路径和安全路径。
6. 输出数据表、路径结果、可视化图、PPT 和汇报手稿。

## 三、数据来源与数据说明

### 1. 真实地图数据

地图道路数据来自高德 Web 服务路径规划接口。项目通过 `src/amap_fetcher.py` 获取起点到终点的真实驾车路线，并将高德返回的导航 step 转换为节点表和道路边表。

高德数据提供：

- 道路名称；
- 道路距离；
- 路线坐标；
- 导航指令；
- 交通状态。

### 2. 真实历史灾害数据

历史灾害数据记录在 `data/historical_disasters/disaster_events.csv`。

地震场景采用 1679 年三河-平谷地震公开历史资料。该事件属于北京及周边地区重要历史地震事件，公开资料记载北京地区出现强烈震感和建筑损坏。本项目将其作为地震影响区案例。

洪水场景采用 2012 年 7 月 21 日北京特大暴雨资料。该事件造成北京城区和郊区大范围暴雨洪涝、内涝和交通影响。本项目将其作为洪水/内涝影响区案例。

### 3. 数据精度说明

真实历史灾害资料通常能提供灾害时间、区域、强度和影响说明，但很少直接提供“某一条道路是否塌方或积水”的逐路段官方记录。因此，本项目采用课程项目中更合理的处理方法：

```text
真实历史灾害事件 -> 灾害影响区缓冲区 -> 与真实道路叠加 -> 得到危险路段
```

这比直接按道路名称指定危险路段更合理，因为危险路段是由灾害影响区和道路空间位置计算得到的。

## 四、道路网络模型

道路网络被抽象为无向图：

- 节点表示高德路线中的坐标点、起点和终点。
- 边表示相邻节点之间的道路段。
- 每条边包含距离、危险类型、拥堵程度、是否可通行和综合权重。

主要数据文件：

| 文件 | 内容 |
| --- | --- |
| `data/amap/nodes.csv` | 高德真实道路节点 |
| `data/amap/edges.csv` | 高德真实道路边 |
| `data/historical_disasters/disaster_events.csv` | 历史灾害事件 |
| `data/amap_earthquake/road_disaster_mapping.csv` | 地震影响区与道路叠加结果 |
| `data/amap_flood/road_disaster_mapping.csv` | 洪水影响区与道路叠加结果 |

## 五、危险路段识别方法

项目先读取历史灾害事件表。每条事件记录包含灾害类型、时间、经纬度、影响半径、危险类型和数据来源。

然后对每条高德道路边进行判断：

1. 取道路边的起点和终点坐标。
2. 将道路边近似为一条线段。
3. 计算灾害影响中心到该线段的最短距离。
4. 如果最短距离小于影响半径，则认为道路进入灾害影响区。
5. 地震场景标记为 `collapse`，洪水场景标记为 `flood`。
6. 未进入灾害影响区但交通状态较差的道路标记为 `congestion`。

## 六、权重设计

普通最短路径只考虑道路距离：

```text
distance_weight = distance
```

安全路径使用综合权重：

```text
safe_weight = distance * risk_factor * (1 + congestion_weight * congestion)
```

| 类型 | 含义 | 风险系数 |
| --- | --- | ---: |
| normal | 正常通行 | 1.0 |
| congestion | 拥堵 | 1.4 |
| flood | 积水 | 2.2 |
| collapse | 塌方 | 4.0 |

风险系数越大，算法越倾向于绕开该路段。

## 七、Dijkstra 算法

Dijkstra 算法是一种经典的最短路径算法，适合边权重非负的图。本项目中，道路距离和综合权重都不是负数，所以可以使用 Dijkstra。

算法过程如下：

1. 起点代价设为 0，其他节点设为无穷大。
2. 每次选择当前总代价最小的节点。
3. 检查该节点连接的道路。
4. 如果经过某条道路到达相邻节点的总代价更小，就更新该节点。
5. 重复直到到达终点。
6. 根据前驱节点回溯得到完整路线。

项目运行两次 Dijkstra：

- 第一次使用 `distance`，得到普通最短路径。
- 第二次使用 `safe_weight`，得到安全路径。

## 八、实验结果

| 场景 | 路径类型 | 总距离 | 综合代价 | 危险类型 | 结果说明 |
| --- | --- | ---: | ---: | --- | --- |
| 地震 | 普通最短路径 | 19.973 km | 19.973 | collapse, congestion | 距离短，但经过历史地震影响区 |
| 地震 | 安全路径 | 26.816 km | 32.209 | congestion | 距离更长，但绕开主要塌方风险 |
| 洪水 | 普通最短路径 | 19.973 km | 19.973 | congestion, flood | 距离短，但经过历史洪水影响区 |
| 洪水 | 安全路径 | 26.816 km | 32.209 | congestion | 距离更长，但绕开主要积水风险 |

结果表明，普通最短路径虽然距离更短，但在灾害场景下可能进入危险区域。安全路径通过提高灾害影响路段的权重，主动选择绕行路线，更适合作为救援推荐路径。

## 九、可视化结果

项目生成两类路径图：

- `outputs/amap_earthquake/route_map_context.png`：地震场景道路背景增强图。
- `outputs/amap_flood/route_map_context.png`：洪水场景道路背景增强图。

图中蓝色粗线表示普通最短路径，绿色粗线表示安全路径。红色或蓝色半透明区域表示历史灾害影响区，红色细线表示塌方风险路段，浅蓝细线表示积水风险路段，橙色细线表示拥堵路段。

## 十、项目成果

本项目最终形成：

1. 数据包：真实道路数据、历史灾害事件表、道路灾害映射表。
2. 源码：高德数据获取、历史灾害映射、Dijkstra 路径规划、可视化绘图。
3. 技术方案：说明数据来源、建模方法、权重设计、算法过程和结果分析。
4. 汇报 PPT：展示真实数据来源、灾害映射方法、路径对比和可视化结果。
5. 汇报手稿：用于课堂讲解和答辩说明。

## 十一、总结

本项目将真实地图数据和公开历史灾害资料结合起来，利用空间叠加方法识别危险道路，再用 Dijkstra 算法计算救援路径。项目说明了在灾害救援场景中，最短路径不一定最适合救援；安全路径虽然可能更远，但能够绕开主要灾害影响区，更符合应急响应需求。
